### 📄 6. .gitlocal

本文探讨了在开发命令行工具时，如何安全地管理包含敏感信息的本地配置文件，并指出了当前 Git 版本控制系统在此场景下的一个设计空白。

作者在构建一个将敏感数据记录在点文件夹（如 `.tool/`）中的 CLI 工具时，发现 Git 缺乏一种机制，能让工具开发者主动声明其生成的文件不应被提交到版本库。这促使他思考并提出了一个潜在的解决方案。

*   **核心问题**：Git 现有的 `.gitignore` 机制是面向项目仓库的，由仓库维护者配置。工具开发者无法确保自己工具生成的本地文件（尤其是包含密钥、令牌等敏感信息的文件）不会被用户意外提交。
*   **现有方案的不足**：虽然可以通过文档告知用户手动将这些路径添加到 `.gitignore`，或使用 `git update-index --assume-unchanged` 等命令，但这些方法都依赖用户操作，不可靠且容易出错。
*   **提出的构想**：作者设想了一个名为 `.gitlocal` 的文件（或类似机制），由工具在本地目录中自动创建。Git 可以识别此文件，并自动忽略其中列出的模式，从而为工具提供一种声明式的方法来保护本地敏感数据。

这一构想旨在填补工具开发工作流中的安全漏洞，推动版本控制系统更好地支持现代开发工具的需求。
