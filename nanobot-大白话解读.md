# Nanobot 大白话技术解读

> 把 Nanobot 想象成一个 **7x24 小时在线的私人助理**，它住在你的电脑/服务器上，能同时接你微信、Telegram、Discord 等各种聊天软件的消息，帮你干活。

---

## 一、Nanobot 是什么？

**一句话**: 一个开源的 AI 聊天机器人框架，你可以把它部署在自己的电脑上，连上各种聊天软件，让 AI 帮你回消息、查资料、写代码、定闹钟。

**类比**: 想象你雇了一个助理（AI），给了他一部手机，手机上装了微信、Telegram、Slack 等等。不管别人从哪个 App 给你发消息，这个助理都能看到并帮你处理。

**为什么叫"nano"**: nano 是"极小"的意思。整个项目核心代码只有 4000 行——作为参考，它的"老大哥" OpenClaw 有好几万行。代码少意味着容易理解、容易改。

---

## 二、整体怎么运作的？

用一个**餐厅**来类比整个系统：

```
顾客（用户）                    餐厅系统（Nanobot）
──────────                    ─────────────────
                              ┌──────────────┐
用户在 Telegram 发消息 ───────>│  前台（Channel）│ 确认是不是VIP客人
                              └──────┬───────┘
                                     │ 把订单写好
                                     ▼
                              ┌──────────────┐
                              │ 传菜窗口（Bus）│ 把订单传给后厨
                              └──────┬───────┘
                                     │
                                     ▼
                              ┌──────────────┐
                              │  大厨（Agent） │ 看订单、想怎么做
                              │              │ 需要食材就去拿
                              │              │ （调用工具）
                              │              │ 做好了把菜传出去
                              └──────┬───────┘
                                     │
                                     ▼
                              ┌──────────────┐
                              │ 传菜窗口（Bus）│ 把做好的菜传回前台
                              └──────┬───────┘
                                     │
                                     ▼
                              ┌──────────────┐
用户在 Telegram 收到回复 <────│  前台（Channel）│ 端给对应的顾客
                              └──────────────┘
```

翻译成技术语言：
- **前台 (Channel)** = 连接各聊天平台的模块
- **传菜窗口 (MessageBus)** = 消息队列，负责在前台和后厨之间传递消息
- **大厨 (AgentLoop)** = 核心 AI 大脑，负责思考和调用工具

---

## 三、六大核心模块详解

### 模块1: 消息渠道 (Channel) — "前台接待"

**干嘛的**: 连接各种聊天软件，把用户消息收进来，把 AI 回复发出去。

**支持哪些平台**:
- 国际: Telegram、Discord、WhatsApp、Slack、Email、Matrix
- 国内: 飞书、钉钉、QQ、Mochat
- 本地: 命令行（直接在终端打字对话）

**怎么工作的**:
1. 你在 Telegram 发了一条 "明天天气怎么样"
2. Telegram Channel 收到消息
3. 先检查你是不是在"白名单"里（安全检查，防止陌生人用你的机器人）
4. 通过之后，把消息打包成标准格式，丢进消息队列
5. AI 处理完后，结果从队列出来，Telegram Channel 再把回复发给你

**关键点**: 每个渠道都是**独立的插头**。你想接 Telegram 就配置 Telegram，想接飞书就配置飞书，互不干扰。

---

### 模块2: 消息总线 (MessageBus) — "传菜窗口"

**干嘛的**: 在渠道和 AI 大脑之间传递消息。

**有多简单**: 整个模块只有 **45 行代码**。就是两个队列：
- **进站队列**: 用户发来的消息排队等 AI 处理
- **出站队列**: AI 的回复排队等渠道发出去

**类比**: 就像餐厅的传菜窗口——厨师不需要知道菜是哪桌点的，前台不需要知道菜是怎么做的。窗口让两边解耦。

---

### 模块3: Agent Loop — "AI 大厨"

**干嘛的**: 这是整个系统最核心的部分——AI 的"大脑"。它接收用户消息，思考，调用工具，得到结果，再思考，最终给出回答。

**工作流程（ReAct 循环）**:

想象你问助理："帮我查一下北京明天的天气，然后帮我设个明早7点的提醒。"

```
轮次1: AI 思考 → "用户要查天气，我需要搜索"
       AI 调用工具: web_search("北京明天天气")
       工具返回: "北京明天晴，最高25度..."

轮次2: AI 思考 → "天气查到了，还需要设提醒"
       AI 调用工具: cron_add("明早7点提醒天气")
       工具返回: "提醒已设置"

轮次3: AI 思考 → "两个任务都完成了，可以回复了"
       AI 回复: "北京明天晴天，最高25度。已帮你设好明早7点的提醒~"
       （没有再调用工具 → 循环结束）
```

**关键规则**:
- 最多循环 **40 轮**（防止 AI 陷入死循环）
- 每轮都是：发给 AI → AI 回复 → 如果有工具调用就执行 → 把结果告诉 AI → 下一轮
- 如果 AI 的回复里**没有工具调用**，说明它觉得可以回答了，循环结束

---

### 模块4: 工具系统 (Tools) — "厨房里的各种厨具"

**干嘛的**: AI 光靠"想"是不够的，它需要工具来和真实世界交互。

**内置工具一览**（用大白话解释）:

| 工具 | 大白话 | 比喻 |
|------|--------|------|
| `read_file` | 读文件 | 像打开一本书看内容 |
| `write_file` | 写文件 | 像在纸上写东西 |
| `edit_file` | 改文件 | 像用橡皮擦掉一行重写 |
| `list_dir` | 看文件夹里有什么 | 像看抽屉里有哪些文件 |
| `exec` | 执行命令行命令 | 像在终端里敲命令 |
| `web_search` | 搜索网页 | 像用百度/Google 搜索 |
| `web_fetch` | 抓取网页内容 | 像打开一个网页读里面的文字 |
| `send_message` | 给指定聊天发消息 | 像主动给某人发微信 |
| `cron` | 设定时任务 | 像设闹钟，到点提醒 |
| `spawn_task` | 创建后台子任务 | 像派一个助手去做耗时的事 |

**安全防护**: `exec` 工具（执行命令）有**黑名单**机制。像 `rm -rf`（删除全部文件）、`shutdown`（关机）这类危险命令会被自动拦截，防止 AI 搞破坏。

**MCP 工具**: 除了内置工具，还可以接入外部工具服务器（MCP 协议）。就像厨房可以叫外卖一样——自己没有的食材，可以从外面现点现送。

---

### 模块5: 记忆系统 (Memory) — "助理的笔记本"

**核心问题**: AI 的"脑容量"有限（上下文窗口），聊太多就会忘记前面的内容。怎么办？

**解决方案——三层笔记本**:

```
第一层: 便签纸（Session History）
  - 就是最近的聊天记录
  - 直接塞给 AI 看
  - 问题: 聊多了放不下

第二层: 记事本（MEMORY.md）
  - 长期重要信息的摘要
  - 比如："用户叫小明，在北京工作，喜欢Python"
  - 每次对话都会看这个本子
  - 人类也能直接打开看/改

第三层: 日记本（HISTORY.md）
  - 按时间记录所有重要事件
  - 格式: "[2026-02-28 10:30] 帮用户查了北京天气并设了闹钟"
  - 可以搜索: AI 需要回忆时可以翻这个日记
```

**记忆整合是怎么发生的**:

当聊天记录超过 100 条时，系统会自动：
1. 把旧的聊天记录交给 AI 读一遍
2. AI 总结出重要信息，写入"记事本"（MEMORY.md）
3. AI 写一段摘要，追加到"日记本"（HISTORY.md）
4. 标记这些旧消息已处理，下次不用再看

**好处**: AI 永远记得重要的事，但不会被太多历史消息撑爆。而且 MEMORY.md 是普通文本文件，你随时可以打开看看 AI 记住了什么，甚至自己改。

### 模块6: 技能系统 (Skills) — "助理的培训手册"

**干嘛的**: 告诉 AI 怎么做某类特定的事情。

**类比**: 就像你给新入职的助理一本手册："如果客户问天气，按这个流程操作..."

**怎么创建一个技能**: 不需要写代码！只需要创建一个 Markdown 文件：

```markdown
<!-- ~/.nanobot/workspace/skills/my-skill/SKILL.md -->

# 翻译技能

当用户要求翻译时:
1. 先确认源语言和目标语言
2. 用 web_search 搜索专业术语
3. 给出翻译结果，附上术语解释
```

**AI 怎么知道有哪些技能**: 每次对话时，AI 的"说明书"里会附上技能列表的摘要。如果需要用某个技能，AI 会自己去读完整的 SKILL.md 文件。

**内置了 9 个技能**: 记忆管理、GitHub 操作、天气查询、社区技能下载、终端管理、定时任务、技能创建器、文本摘要等。

## 四、LLM Provider — "AI 的大脑从哪来"

Nanobot 自己不包含 AI 大脑——它需要连接外部的 AI 服务（就像手机壳不包含手机，但可以套在各种品牌的手机上）。

**支持 16 个 AI 服务商**:

| 类别 | 服务商 | 说明 |
|------|--------|------|
| 推荐 | OpenRouter | 一个 key 用所有模型，就像"AI 界的万能充电器" |
| 国际 | Anthropic (Claude), OpenAI (GPT), Google (Gemini), DeepSeek | 各家官方直连 |
| 国内 | 通义千问, 智谱, 月之暗面(Kimi), MiniMax, 硅基流动, 火山引擎 | 国内服务商 |
| 本地 | vLLM | 在你自己电脑上跑模型，不用花钱 |
| 特殊 | OpenAI Codex, GitHub Copilot | 用 OAuth 登录，不需要 API key |

**怎么切换 AI**: 只需要改一行配置：

```json
{
  "agents": {
    "defaults": {
      "model": "anthropic/claude-opus-4-5"
    }
  }
}
```

改成 `"deepseek/deepseek-chat"` 就切换到 DeepSeek 了。系统会自动识别该用哪个服务商。

**省钱功能 (Prompt Caching)**: 对于 Anthropic 等支持的服务商，系统会标记哪些内容是"不变的"（比如 system prompt），这样 AI 服务商可以缓存这部分，下次对话时不用重新计算，**省钱省时间**。

## 五、辅助服务

### 定时任务 (Cron) — "闹钟"

你可以让 AI 帮你设定时任务，比如：
- "每天早上8点给我发一条天气预报"
- "下午3点提醒我开会"
- "每周一检查一下 GitHub 上的新 issue"

支持三种模式：
- **一次性**: "30分钟后提醒我"
- **循环**: "每2小时提醒一次"
- **Cron 表达式**: "每周一到周五早上9点" (`0 9 * * 1-5`)

### 心跳 (Heartbeat) — "自动巡逻"

每 30 分钟，系统会自动醒来看一下 `HEARTBEAT.md` 文件里有没有待办事项。如果有，就自动执行。

```markdown
<!-- HEARTBEAT.md -->
## 周期任务
- [ ] 检查天气预报，有极端天气就提醒我
- [ ] 扫描邮箱有没有紧急邮件
```

就像你给助理一个**每日待办清单**，他每半小时看一眼有没有需要处理的。

### 子Agent (Subagent) — "派小弟去干活"

当主 AI 遇到耗时的任务时，可以派一个"分身"在后台处理，不阻塞主对话：

```
你: "帮我分析一下这份100页的报告"
AI: "好的，我派了一个后台任务去分析，完成后会通知你。你可以继续跟我聊其他的。"
...（你继续聊别的）...
AI: "刚才的报告分析完了，主要结论是..."
```

## 六、安全机制 — "防止 AI 搞事"

| 防护 | 说明 | 大白话 |
|------|------|--------|
| 白名单 | 只有你指定的用户才能跟机器人聊 | "只有VIP才能进" |
| 命令黑名单 | 危险命令（删除、格式化、关机）自动拦截 | "不让AI碰危险按钮" |
| 文件夹限制 | 可以限制AI只能操作指定文件夹 | "只能在自己工位上动东西" |
| 路径检查 | 防止AI通过 `../../` 跳出限制 | "不让走后门" |
| 错误隔离 | AI出错的回复不保存到历史 | "坏的菜不上桌" |
| 超时保护 | 命令执行超时自动终止 | "做太久就停下来" |

## 七、配置 — "怎么设置"

所有配置都在一个文件里: `~/.nanobot/config.json`

最简配置（只需3行就能用）：
```json
{
  "providers": {
    "openrouter": {
      "apiKey": "sk-or-v1-你的密钥"
    }
  }
}
```

想开 Telegram？加几行：
```json
{
  "channels": {
    "telegram": {
      "enabled": true,
      "token": "你的Bot Token",
      "allowFrom": ["你的用户ID"]
    }
  }
}
```

就这么简单。不需要什么数据库、不需要配环境变量、不需要写代码。

## 八、部署 — "怎么跑起来"

### 最简方式（3步）

```bash
pip install nanobot-ai       # 1. 安装
nanobot onboard              # 2. 初始化
# 编辑 ~/.nanobot/config.json 填入 API key
nanobot agent                # 3. 开始聊天！
```

### 想 24 小时在线

```bash
nanobot gateway              # 启动网关模式，连接所有聊天平台
```

### 用 Docker（不想装 Python）

```bash
docker compose up -d nanobot-gateway   # 一行命令启动
```

### 开机自启（Linux 服务器）

配一个 systemd 服务，服务器重启后自动恢复运行。

## 九、Nanobot vs OpenClaw — 快速对比

| | Nanobot | OpenClaw |
|---|---|---|
| **体量** | 小巧（1万行代码） | 庞大（数万行） |
| **语言** | Python | TypeScript |
| **类比** | 瑞士军刀 | 全套工具箱 |
| **上手难度** | pip install 就能用 | 需要 pnpm、Node.js 等 |
| **手机App** | 没有 | 有 iOS/Android/macOS |
| **国内平台** | 飞书/钉钉/QQ | 没有 |
| **安全级别** | 基础（适合个人用） | 企业级（进程隔离、审批） |
| **适合谁** | 个人用户、研究者、极客 | 需要多端体验的团队 |

## 十、总结

**Nanobot 的核心价值**:

1. **简单**: 4000 行核心代码，一个 JSON 配置文件，pip install 即用
2. **全渠道**: 12 个聊天平台，一套 AI 全部搞定
3. **全模型**: 16 个 AI 服务商随意切换，改一行配置的事
4. **有记忆**: 三层记忆系统，聊多少都不会忘重要的事
5. **能干活**: 搜网页、读写文件、执行命令、设闹钟、后台任务
6. **安全**: 白名单 + 命令黑名单 + 文件夹限制
7. **可扩展**: 新技能写个 Markdown 就行，新工具几十行代码

**一句话总结**: Nanobot 就是一个你可以自己部署、自己控制的 AI 助手，比 ChatGPT 多了"能干活"（工具调用）和"全渠道"（各种聊天软件），比 OpenClaw 简单 10 倍。
