我书架上有不少书，买的时候都想着会看。买完放在那，大多数没翻过。

收藏夹里也攒了几百本"想看的书"。偶尔翻到，心里会有点堵。说不上是愧疚还是焦虑，就是觉得自己好像一直在"想"看书，但从来没真的开始。

做 AI 之后，这种感觉更明显。每天在追新的工具、新的模型、新的玩法，脑子一直是满的。坐下来安安静静翻本书，好像成了一件很奢侈的事。

春节放假，事少了，人也闲下来了。我想，能不能用 AI 做个拆书的工具。先看拆解，觉得好再去找原书。至少比放在收藏夹里强。

---

## 提示词的阶段

最早是写 prompt。把一本书的信息给 AI，让它输出一篇读书笔记。

实际做起来才知道，这件事比想的要难。

格式一直不稳定，加粗到处都是，标题层级也乱。字数也不好控制，说好5000字的笔记，经常写到3000字就开始凑。

但最困扰我的是内容。AI 会把拆解用到的分析方法直接写进正文。"运用第一性原理分析""通过 DIKW 框架可以看出"——读起来像一篇学术报告，不像一个人在跟你聊一本书。

我其实就想要一种感觉：一个读过这本书的朋友，用自己的话，跟你说说这本书讲了什么。

为了靠近这个感觉，我开始一点一点加规则。先设计了拆解流程，定义每个阶段该写什么、重点是什么。然后补各种约束：这个部分不少于200字，正文里不能出现分析方法的名称，加粗只能在核心观点首次提出时用。

就这样一版一版改，prompt 慢慢涨到了4万多字。

4万字的提示词，就为了让 AI 好好写篇读书笔记。现在想想有点荒谬，但当时确实不知道还能怎么办。效果也说不上多好，总是这边调好了，那边又出问题。一直在补。

## 从提示词到 skill

后来 Claude Code 出了 skill 功能。对我来说，这不只是换了个工具，而是做事方式的一次变化。

之前写 prompt 的时候，我既是设计者也是执行者。每次跑完自己看效果，有问题就改 prompt，改完再跑。所有事情都压在自己身上。

用 skill 之后，我的角色慢慢变成了需求方和验收方。我说我想要什么样的效果，Claude Code 来设计流程、拆分模块、写验证脚本。我负责去读、去用、去反馈哪里还不够好。

这个变化挺大的。

结构上也清晰了很多。prompt 是一大块文字，改多了自己都理不清。skill 把事情拆开了，主文件管流程，references 放范例和风格指南，scripts 放检测脚本，改哪里就动哪里。

以前看生成的笔记有没有问题，全靠自己一篇篇读。现在有验证脚本自动查：字数、格式、有没有出现不该有的 AI 用语。

做完这些之后我有一个感受，可能也是这件事里最重要的一个认知：**把流程标准化，比把提示词写到完美有用得多。** prompt 写到4万字，也不如一套结构清晰的流程。

## 一版一版调

换成 skill 之后，迭代的节奏快了。

很多反馈是老板给的。她自己爱看书，拿着生成的笔记逐条看，说的话也很直。

"太长了。"

"这段不像人写的。"

"我想知道这本书讲了什么，不是你用了什么分析方法。"

每条反馈后面都会跟着一次修改。

慢慢地加上了10来种拆解方法，不过全部藏在内部逻辑里，读者在正文中不会看到。17种书的类型也做了区分，文学和经济学的拆法自然不同。验证脚本从十来项扩展到了58项。

现在的状态是，跑10本书，一次性生成，58项检测全部通过，基本不需要人工干预。

不过也不是没有遗憾。加粗滥用的问题还是会偶尔冒出来，到现在没有完全解决。有些书的拆解读起来还是偏工具化，少了一种真正在跟人聊这本书的感觉。

每本书都是一个故事。用一个通用的框架去套所有故事，总归是差了点什么。这是后面想继续做的事。

## 没想到的事

做这个工具的时候，出发点就是省时间。不用读原书也能了解内容，多好。

但264本拆完之后，有一个我没预料到的变化：我开始看书了。

不是每本都看。工具方法类的书，拆解看完就够了，核心观点记住就行。

但有些书不一样。

有一次读《活着》的拆解，里面引了有庆赤脚跑步的那段。就几句话，但我读完停了很久。后来把原书找出来了。

还有一次是《金刚经》。拆解里写到"一切有为法，如梦幻泡影"。这句话在朋友圈见过很多次，每次都是划过去。但放在那篇拆解的上下文里读到，感觉不太一样。

那天晚上我找了原文，从头看了一遍。

做 AI 这段时间，一直在追快的东西。工具在更新，模型在迭代，手上的事也停不下来。脑子转得挺快，但好像很少有安静的时候。

看书的时候可以安静一会儿。

**这大概是做这个工具最意外的收获。不是264本读书笔记，是我自己开始看书了。**

---

那些书就放在那里。想看的时候先翻翻拆解，碰到真正想读的，就慢慢看。

不急。

AI 让一切都可以很快。但有些东西，就是得慢一点。

偶尔看看书，对自己挺好的。
